# Issue 03: Make AI Rule Selection Timeout Configurable

**Severity:** MEDIUM
**Category:** Interception & Routing Logic

---

## Problem

In `src/server/features/rules/ai/ai-choose-rule.ts` at line 68, the AI rule selection timeout is hardcoded to `3500` milliseconds. When the LLM exceeds this timeout, the system falls back to keyword heuristics (Issue 01). Even after Issue 01 removes the heuristics, a short timeout means emails may go unclassified during high load.

```typescript
// Line 68 in ai-choose-rule.ts
3500,
```

This value is not configurable per user, per account, or per environment.

---

## Root Cause

The timeout was set as a single constant during initial development. No infrastructure exists for per-account or per-environment configuration of rule processing parameters.

---

## Step-by-Step Fix

### Step 1: Add a column to the EmailAccount model for AI timeout preference

**File:** `prisma/schema.prisma`

Find the `EmailAccount` model and add an optional field:

```prisma
model EmailAccount {
  // ... existing fields ...
  aiRuleTimeoutMs  Int?  // Per-account AI rule selection timeout in ms. Null = use default.
}
```

### Step 2: Create a Prisma migration

Run:

```bash
bunx prisma migrate dev --name add_ai_rule_timeout
```

This creates a new migration file in `prisma/migrations/` with the ALTER TABLE statement.

### Step 3: Define a default timeout constant and environment variable

**File:** `src/server/features/rules/ai/ai-choose-rule.ts`

At the top of the file (near the imports), add:

```typescript
import { env } from "@/server/lib/env";

/**
 * Default AI rule selection timeout in milliseconds.
 * Can be overridden per-account via emailAccount.aiRuleTimeoutMs
 * or globally via AI_RULE_TIMEOUT_MS environment variable.
 */
const DEFAULT_AI_RULE_TIMEOUT_MS = 10_000; // 10 seconds

function getAiRuleTimeout(emailAccount: { aiRuleTimeoutMs?: number | null }): number {
  // Priority: per-account > env var > default
  if (emailAccount.aiRuleTimeoutMs && emailAccount.aiRuleTimeoutMs > 0) {
    return emailAccount.aiRuleTimeoutMs;
  }
  const envTimeout = env.AI_RULE_TIMEOUT_MS;
  if (envTimeout && Number(envTimeout) > 0) {
    return Number(envTimeout);
  }
  return DEFAULT_AI_RULE_TIMEOUT_MS;
}
```

### Step 4: Replace the hardcoded timeout

**File:** `src/server/features/rules/ai/ai-choose-rule.ts`

Find line 68 where the timeout is used (inside a `Promise.race` or `setTimeout` call). Replace:

```typescript
// Before:
3500,

// After:
getAiRuleTimeout(emailAccount),
```

Make sure the `emailAccount` parameter is in scope where the timeout is used. It is already passed to `aiChooseRule` as a parameter.

### Step 5: Add the env var to the env schema (if applicable)

**File:** `src/server/lib/env.ts` (or wherever environment variables are validated)

Add `AI_RULE_TIMEOUT_MS` as an optional string/number field:

```typescript
AI_RULE_TIMEOUT_MS: z.string().optional(),
```
